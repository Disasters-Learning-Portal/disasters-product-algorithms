# Project Rules for AI Assistants

NASA Disasters product-algorithms repo: GeoTIFF → COG pipelines with S3 upload, consumed downstream by `veda-data-airflow`'s `build_stac` handler.

## Hard rules

1. **Default `target_crs` / `dst_crs` is `EPSG:3857`** (Web Mercator). All notebook templates, `cog_utils.convert_to_cog`, `main_processor.convert_to_cog`, and `SimpleProcessor` default to it. Reason: airflow's `rio_stac.get_dataset_geom` crashes on EPSG:4326 outputs because modern PROJ writes the WGS 84 ensemble WKT + lat-first axis, producing `"Point outside of projection domain"`. Web Mercator sidesteps both. **Validated end-to-end** against veda-data-airflow's `build_stac` on a global Mollweide source (GAIA `total.tif`, reprojected with `needs_webmerc_clip()` auto-clip enabled) — the DAG accepted the output cleanly. **Do not change the default back to EPSG:4326 without solving the ensemble + axis problem.**

2. **Do not reintroduce a `normalize_wgs84_crs()` helper.** The old `gdal_edit.py -a_srs '+proj=longlat +datum=WGS84 +no_defs'` approach was deleted because PROJ re-canonicalizes the WKT back to the WGS 84 ensemble on read — the edit looked successful in `gdal_edit` output but the next consumer (`rio_stac.get_dataset_geom`) still saw ensemble + lat-first axis. The fix that actually works is **target EPSG:3857 in the first place** (rule #1 above). If you need to ship to EPSG:4326 specifically, you'll need a different mitigation (e.g. axis-order rewrite via low-level libtiff edit) — but don't add it back without a real test that proves the downstream consumer accepts it.

3. **Treat `DST_CRS`/`target_crs` as `None` when value is `None` / `""` / `"None"` (any case).** Means: preserve native CRS, skip the warp step entirely. Do not pass `--dst-crs None` to `rio warp` (OGR error 5).

4. **If the source `.tif` has no embedded CRS, warn and skip reprojection.** Don't try to warp a CRS-less file.

5. **All temp files go to `/tmp/`** (never `./s3_temp` or any project-relative path). Don't reintroduce relative temp paths.

6. **GDAL must come from conda, not pip.** dylib version mismatches otherwise. Don't add `gdal` to pip requirements.

7. **Notebooks don't share kernel state across cells reliably** — testing-notebooks cells that import from `shared_utils` must contain their own `repo_root` discovery block, not rely on a "setup" cell having been run first.

8. **All raster hot paths use `NUM_THREADS=ALL_CPUS`.** `gdalwarp` calls include `-multi -wo NUM_THREADS=ALL_CPUS --config GDAL_NUM_THREADS ALL_CPUS`; `rio cogeo create` calls include `--co NUM_THREADS=ALL_CPUS`; `rasterio.warp.reproject` calls pass `num_threads=os.cpu_count()` (it needs an int, not the GDAL `ALL_CPUS` string). Don't drop these flags.

9. **Do not create `environment.yml` in this repo.** `jupyter-repo2docker` builds the Disasters Hub image from the `pangeo-notebook-veda-image` checkout — anything here is ignored. For new deps that need to land in the hub image:
   - **Pip-installable** → add to `[project] dependencies` in `pyproject.toml`. Push to main. Done. (Flows in via the `pip: - git+https://...algorithms.git` line in image repo's `environment.yml` + `ALGORITHMS_SHA` cache-buster.)
   - **Conda-only** → add a line to `hub-conda-deps.txt`. The `sync-conda-deps.yml` workflow auto-opens a PR in `pangeo-notebook-veda-image` updating its managed block. Click-merge that PR; next image build picks it up.

   The conda-dep comment block in `pyproject.toml` is the DEV-LOCAL install spec (what to conda-install on your laptop), NOT what the image installs. Do not conflate. Full flow in `docs/HUB_DEPLOYMENT.md`.

10. **`process_*` CLIs missing on a fresh hub pod is an image-build bug, not a local-install bug.** The image-repo Dockerfile's `RUN conda env update` layer is cached on `environment.yml` contents; the algorithms install line is unpinned, so without the `ARG ALGORITHMS_SHA` cache-buster (passed from the dispatch payload by `pangeo-notebook-veda-image/.github/workflows/build-and-push*.yaml`), pushes here never re-trigger the pip install. **Check the image-repo build log and Dockerfile before suggesting `pip install -e .` as a fix.** That workaround stays valid for hot-iterating local edits inside one hub session, never for "every fresh pod is broken."

## Patterns to follow

- **One filename module**: `shared_utils/file_naming.py` is the single source of truth for `extract_datetime_from_filename`, `categorize_file`, `create_output_filename`, `no_change`, `DATETIME_PATTERNS`. Notebooks should `from shared_utils.file_naming import ...`, not redefine. The earlier legacy helpers (`extract_date_from_filename`, `convert_date`, `parse_filename_components`, `create_cog_filename`, `create_output_path`) were deleted — their behavior is covered by the new API plus plain `os.path.join`. If you find them referenced in old code, replace with the new helpers.
- **One Web-Mercator-clip detector**: `shared_utils/reprojection.needs_webmerc_clip(src_or_path, dst_crs)` returns True iff the source's geographic lat bounds exceed ±85.05° AND `dst_crs` ≈ EPSG:3857. Returns False for the 99% regional-raster case (no behavior change for sensor pipelines). Used by `cog_utils.convert_to_cog` and the CLI notebook.
- **Notebook tiers stay paired**: `notebooks/<x>.ipynb` (Hub/CLI subprocess style, has Quarto YAML) and `notebooks/testing-notebooks/<x>.ipynb` (import-based, for local dev). They should differ only in *style*, not behavior — both pull from the same `shared_utils` helpers.
- New CLI scripts: add entry point under `[project.scripts]` in `pyproject.toml`, add the package to `[tool.setuptools.packages.find].include`. Sensor pipelines and standalone utilities both use the thin `cli.py` exec-wrapper pattern (see `landsat/cli.py` or `raster_tools/cli.py`). Full walkthrough in `docs/ADDING_FUNCTIONS_TUTORIAL.md`.
- **Test fixtures with real data go in `tests/fixtures/`**, kept small (<500KB) and DEFLATE-compressed with `predictor=3` for float32. Crop from production rasters with `rasterio.windows.Window` + `window_transform` to preserve georeferencing. Hard-code the expected min/max/mean in the test so re-cropping forces an explicit update (see `tests/unit/test_summarize_raster.py::test_summarize_raster_on_real_gaia_atlanta`).
- **Tests must import from the submodule, not the package root** — `from shared_utils.cog_utils import X`, not `from shared_utils import X`. The package `__init__.py` chains optional imports including `cog_metadata`, which calls `gdal.UseExceptions()` at module load. On environments with a GDAL python-binding / dylib mismatch (common on dev laptops, see hard rule #6) this raises `AttributeError`, which the `try/except ImportError` in `__init__.py` does not catch — every test that touches the package root then fails at collection. Submodule imports skip the broken chain.
- COG defaults: ZSTD level 22, 512x512 tiles, 5 overview levels. Nodata auto: uint8=0, int16=-9999, float=-9999.0.

## Patterns to avoid

- Don't introduce new `convert_to_cog` symbols in `shared_utils/` — there are already two (`cog_utils.convert_to_cog` and `main_processor.convert_to_cog` aliased as `convert_to_cog_s3`); a third risks star-import shadowing. They have different roles: the former is the local-file warp+COG primitive; the latter is the S3 download → engine → upload orchestrator that delegates to the former. **Don't re-implement warp+COG logic in main_processor — it's a thin wrapper.**
- Don't reintroduce the chunk_config / size-based branching that used to live in `main_processor.convert_to_cog`. It was deleted in the unification refactor because `gdalwarp` + `rio cogeo create` handle their own chunking via `BLOCKSIZE` and `NUM_THREADS=ALL_CPUS`.
- Don't commit `.ipynb_checkpoints/*` or `__pycache__/*` — both are in `.gitignore` but tend to leak in during merges.

## Architecture quick map

- `shared_utils/cog_utils.convert_to_cog(input_tif, output_cog=None, dst_crs='EPSG:3857', resampling_method=None, clip_to_webmerc=None, ...)` — **the engine**. Takes a local path (or `/vsi*` URI), runs `gdalwarp` (with auto-clip when needed) + `rio cogeo create`, writes a local COG.
- `shared_utils/main_processor.convert_to_cog(name, bucket, ..., target_crs='EPSG:3857', stream_from_s3=True, ...)` — **the S3 orchestrator** (~216 lines, was ~520). Probes `/vsis3/...` streaming first, falls back to `/tmp` download, calls `cog_utils.convert_to_cog`, validates, uploads to S3.
- `shared_utils/notebook_helpers.SimpleProcessor` — config-driven class wrapper used by GAIA / simple_disaster templates. Delegates filename/categorization to `shared_utils.file_naming`, delegates conversion to `main_processor.convert_to_cog`. Accepts `target_crs`, `resampling`, `clip_to_webmerc`, `stream_from_s3` config keys.
- `shared_utils/reprojection.py` — pure-rasterio warp helpers + `needs_webmerc_clip()` + `WEBMERC_EXTENT_M` / `WEBMERC_EPSGS` constants.

## Downstream dependency

Output COGs are read by `veda-data-airflow` DAGs (separate repo). The `build_stac` task is the strictest consumer — fix CRS issues here, not there. Two production failure modes to be aware of:
- **WGS 84 ensemble + lat-first axis** — kills `rio_stac.get_dataset_geom`. Mitigated by defaulting to EPSG:3857.
- **Source extent outside target CRS domain** — global Mollweide / polar stereographic warped to EPSG:3857 without a ±85° clip produces 50+ GB nonsense or fails. Mitigated by `needs_webmerc_clip()` auto-detect.
